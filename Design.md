# AbelTools Design Guide

> 문서 상태: 첫 번째 디자인 초안  
> 기준: 2026-07-20까지의 AbelTools 디자인 대화  
> 주의: 현재 구현된 화면은 이 문서의 근거가 아니다. 기존 코드는 기능 실험 및 임시 구현으로 간주하고, 충돌할 경우 이 문서와 이후의 디자인 결정을 우선한다.

## 1. 프로젝트의 인상

AbelTools는 음악 제작자의 반복적이고 번거로운 작업을 줄여 주는 도구 모음이자, 프리셋과 작업 방식을 공유하는 작은 커뮤니티다.

첫인상은 다음 네 단어로 설명할 수 있어야 한다.

- **Modern**: 정보 구조와 기능은 명확하고 정돈되어 있다.
- **Cute**: 사카노, 리본, 손글씨형 로고처럼 기억에 남는 귀여움이 있다.
- **Useful**: 장식보다 실제 음악 작업에 도움이 된다는 인상이 먼저 온다.
- **Personal**: 대형 SaaS보다 제작자가 직접 운영하는 애정 어린 작업실처럼 느껴진다.

귀여움은 모든 요소를 화려하게 꾸미는 방식으로 만들지 않는다. 기본 UI는 차분하고 간결하게 유지하고, 로고·캐릭터·메시지·작은 인터랙션에 귀여움을 집중한다.

## 2. 브랜드 구조

### 2.1 핵심 문장

**Let’s music — Smarter, Better, Funnier.**

문법적으로 완벽한 일반 문장보다 AbelTools만의 장난스럽고 기억하기 쉬운 구호로 사용한다. `music`을 명사이면서 행동처럼 취급하는 것이 의도다.

### 2.2 브랜드 성격

- 음악 기술을 잘 알지만 어렵게 말하지 않는다.
- 친근하지만 기능의 신뢰성을 해치도록 가볍지는 않다.
- 사용자의 작업 방식을 강요하지 않고 선택지를 제공한다.
- 개발자 중심의 표현보다 음악 제작자가 이해하는 표현을 우선한다.
- “생산성”만 강조하지 않고 작업의 재미와 공유의 즐거움도 함께 다룬다.

### 2.3 이름 표기

- 공식 표기: `AbelTools`
- 제품군 표기: `OrchTools`, `ProdTools`, `MyBlog`
- 도구 이름은 현재 기획명을 유지하되 출시 전에 대소문자 규칙을 한 번 통일한다.
- 헤더에서는 전체 워드마크 또는 리본 심벌과 `AbelTools` 조합을 사용한다.

## 3. 컬러 시스템

색은 장식이 아니라 콘텐츠의 역할을 알려 주는 언어로 사용한다.

### 3.1 역할별 색상

| 역할 | 계열 | 사용하는 곳 |
|---|---|---|
| 기본 구조 | 검정·먹색 | 본문, 제목, 내비게이션, 로고, 아이콘, 중요한 윤곽선 |
| 도구와 기능 | 노랑·골드 | OrchTools, ProdTools, 실행 버튼, 유용한 기능, 선택 또는 활성 상태 |
| 사람과 소통 | 연초록 | 개인 메시지, Contributors, Contact, 공유와 커뮤니티 영역 |
| 바탕과 여백 | 크림·백색 | 페이지 배경, 카드, 긴 글을 읽는 영역 |
| 상태 표시 | 별도 의미색 | 오류, 경고, 성공 등 실제 상태 전달에만 제한적으로 사용 |

검정·노랑·초록을 한 컴포넌트에서 같은 비중으로 섞지 않는다. 화면마다 주역이 되는 의미색 하나를 정하고, 나머지는 보조로 사용한다.

### 3.2 초안 팔레트

아래 값은 구현을 시작하기 위한 제안값이며 Figma에서 시각 검토 후 조정할 수 있다.


### 3.3 사용 원칙

- 긴 본문은 항상 충분히 진한 먹색으로 쓴다.
- 연노랑과 연초록 위에 흰색 글자를 사용하지 않는다.
- 노랑은 “기능적이다”, 초록은 “사람과 연결된다”는 의미를 유지한다.
- 회색으로 큰 면을 임시로 채운 디자인은 완성 색상으로 간주하지 않는다.
- 그라디언트는 캐릭터 이미지의 페이드나 미세한 조명 표현 외에는 최소화한다.

## 4. 타이포그래피

### 4.1 기본 방향

둥글고 손으로 쓴 듯한 로고와, 정돈된 UI 글자가 함께 존재한다. 로고 글씨를 본문 전체에 그대로 확장하지 않는다.

- **로고/브랜드 표현**: 현재 제작한 둥근 `AbelTools` 워드마크
- **한국어 UI와 제목**: Maplestory Bold
- **한국어 본문**: Maplestory Light 또는 검증된 가독성 좋은 한글 산세리프
- **영문 긴 본문 및 숫자가 많은 도구 UI**: Nunito 계열을 우선 검토
- **코드, MIDI 값, 포트, 채널, 노트명**: 폭이 일정한 모노스페이스 글꼴 사용 가능

Maplestory는 브랜드의 친근함을 잘 전달하지만 작은 크기의 긴 영문 본문과 정밀한 데이터 표에서는 가독성을 별도로 검증한다.

### 4.2 크기 원칙

- 메인 히어로 제목: 48–80px, 화면에 따라 유동적으로 조절
- 페이지 제목: 40–64px
- 섹션 제목: 28–40px
- 본문: 16–18px
- 도구 UI 본문: 최소 14px
- 보조 설명: 최소 12px, 핵심 정보를 이 크기에만 의존하지 않는다.
- 본문의 줄 간격: 글자 크기의 약 1.55–1.75배

## 5. 로고와 사카노

### 5.1 로고 체계

1. **헤더 로고**: 사카노가 착용한 형태의 귀여운 검은 리본 + `AbelTools` 워드마크
2. **메인 로고**: 둥근 워드마크와 사카노가 결합된 넓은 구성
3. **심벌**: 작은 화면, favicon, 로딩 등에 사용할 검은 리본 단독형

리본은 일반적인 나비넥타이 모양이 아니라 사카노의 머리 위에 있는 크고 각진 헤어 리본의 실루엣을 따른다. 작은 크기에서도 두 개의 큰 날개와 가운데 매듭이 구분되어야 한다.

### 5.2 사카노의 역할

사카노는 단순 장식용 마스코트가 아니라 사이트를 안내하고 브랜드의 개인적인 온도를 전달하는 캐릭터다.

- 금발, 초록색 눈, 큰 검은 리본은 핵심 식별 요소로 유지한다.
- 웹사이트에서는 기존 정장형 복장보다 캐주얼하고 SD 비율이 높은 버전을 기본으로 한다.
- 검정·크림·골드에 소량의 초록 포인트를 사용해 사이트 팔레트와 연결한다.
- 모든 섹션에 반복 노출하지 않는다. 메인 히어로, About, 빈 상태, 완료 피드백처럼 기억에 남을 지점에서 사용한다.

### 5.3 메인 연출

메인 로고 뒤에서 사카노가 담장을 넘겨다보듯 나타나며, 손은 워드마크의 `e`와 `o` 부근에 올린다. 로고는 전경, 캐릭터 몸통은 후경에 두어 겹침이 명확해야 한다.

스크롤 연출을 적용할 경우:

1. 처음에는 워드마크만 안정적으로 보인다.
2. 스크롤 진행에 따라 `e`와 `o`가 가로로 아주 조금 눌린다.
3. 그 틈 또는 뒤쪽에서 사카노의 리본과 머리가 먼저 나타난다.
4. 양손이 글자 위에 자리 잡은 뒤 얼굴과 상체가 자연스럽게 드러난다.
5. 애니메이션이 끝나도 글자를 읽는 데 방해가 없어야 한다.

변형은 장난스러운 탄성을 느낄 정도로만 사용한다. 과도한 찌그러짐, 긴 지연, 스크롤을 가로채는 방식은 사용하지 않는다. `prefers-reduced-motion`에서는 변형 없이 최종 장면을 보여 준다.

### 5.4 About 이미지

- 상체 하단이 딱 잘린 것처럼 보이지 않도록 투명 알파 페이드로 끝낸다.
- 배경색에 흰색을 덧칠해 페이드하지 않는다. 크림, 초록 등 어느 배경에서도 쓸 수 있는 실제 투명 PNG/WebP가 필요하다.
- 얼굴과 손은 작은 화면에서도 잘리지 않도록 안전 영역을 확보한다.

## 6. 레이아웃

### 6.1 기본 구조

당근이나 토스처럼 콘텐츠는 중앙 컨테이너에 집중시키고, 넓은 화면의 좌우는 호흡을 위한 여백으로 남긴다. 배경색과 섹션 밴드는 화면 전체 폭을 사용할 수 있지만, 읽어야 하는 내용까지 끝없이 넓히지 않는다.

- 일반 마케팅 페이지: 최대 1120–1200px
- 긴 글과 메시지: 최대 700–850px
- 복잡한 편집 도구: 최대 1280–1440px 또는 안전한 전체 폭
- 좌우 여백: 모바일 20–24px, 태블릿 32px, 데스크톱 40px 이상
- 섹션 간격: 데스크톱 96–144px, 모바일 64–96px

### 6.2 화면 밀도

- 랜딩 페이지는 큰 여백과 소수의 강한 요소를 사용한다.
- 실제 도구 화면은 정보 밀도를 높일 수 있지만 그룹과 계층을 분명히 한다.
- 카드가 많다는 이유만으로 모든 것을 동일한 카드 모양에 넣지 않는다.
- 화면의 중앙 집중은 “빈 공간을 채우지 못한 것”이 아니라 읽기 폭과 시선 흐름을 통제하기 위한 결정이다.

### 6.3 반응형

- 모바일에서 핵심 기능을 제거하지 않는다.
- 복잡한 데스크톱 편집기는 모바일에서 열람·간단 편집 흐름을 제공하고, 정밀 편집이 어렵다면 그 사실을 명확히 안내한다.
- 내비게이션은 좁은 화면에서 줄바꿈시키기보다 명시적인 모바일 메뉴로 전환한다.
- 고정 폭 이미지와 절대 위치 장식은 텍스트 및 버튼과 겹치지 않아야 한다.

## 7. 페이지별 시각 역할

### 7.1 Home

홈은 기능 목록을 모두 설명하는 페이지가 아니라 AbelTools의 세계관과 가장 중요한 진입점을 보여 준다.

권장 순서:

1. 헤더와 간결한 브랜드 로고
2. 메인 워드마크 + 사카노 피킹 인터랙션
3. “Let’s music — Smarter, Better, Funnier.”
4. 연초록 편지 형태의 개인 메시지
5. OrchTools와 ProdTools의 대표 기능
6. Contributors 또는 공유 참여 유도
7. Footer와 Contact/광고 안내

### 7.2 OrchTools

- BRSOfriend
- KontaktCutter / Audio Tokenizer
- 이후 추가될 오케스트레이션 도구

노랑을 기능색으로 사용하되, 악보·MIDI·키스위치 정보는 검정 중심으로 명확하게 표시한다.

### 7.3 ProdTools

- MixerPalette / Color Palette
- EffectChainShare
- 이후 추가될 프로덕션 도구

OrchTools와 같은 제품 계열이므로 노랑 기반을 공유한다. 각 도구의 고유 색상은 식별용 보조색으로만 사용한다.

### 7.4 Contributors와 Contact

연초록을 대표색으로 사용한다. 이름, 연락처, 기여 내용과 일자가 읽기 쉽게 연결되어야 한다. Contributors는 단순 명단보다 각자가 무엇을 공유했는지 보여 주는 작은 프로필 아카이브에 가깝게 만든다.

Contact는 광고 및 협업 문의를 숨기지 않고 명확하게 안내한다. 친근한 문구를 사용하되 실제 연락 방식과 기대 응답 범위는 구체적으로 쓴다.

### 7.5 MyBlog

MySong, MyTips, History는 제품 UI보다 편집 디자인에 가깝게 구성한다. 기본 배경은 크림과 먹색으로 차분하게 유지하고, 글의 분류 태그에만 노랑 또는 초록을 선택적으로 사용한다.

## 8. 핵심 컴포넌트

### 8.1 헤더

- 왼쪽: 검은 리본 + AbelTools 워드마크
- 오른쪽: 주요 제품군, Contributors, Contact, 언어 선택
- 기본 상태는 가볍고 배경과 자연스럽게 연결한다.
- 스크롤 후 고정할 경우 얇은 경계선 또는 약한 반투명 배경만 사용한다.
- 드롭다운은 키보드와 터치에서도 열 수 있어야 하며 hover에만 의존하지 않는다.

### 8.2 버튼

- Primary: 먹색 바탕 또는 기능 맥락의 노랑 바탕
- Secondary: 종이색 바탕 + 진한 테두리
- Communication CTA: 연초록 바탕
- 모서리는 둥글게 하되 모든 버튼을 과도한 캡슐형으로 만들 필요는 없다.
- 아이콘만 있는 버튼에는 tooltip과 접근 가능한 이름이 필요하다.
- 눌림, hover, focus, disabled 상태를 모두 설계한다.

### 8.3 카드와 패널

- 마케팅 카드: 넉넉한 여백, 둥근 모서리, 약한 선
- 도구 패널: 더 작은 radius와 높은 정보 밀도
- 기본 그림자는 거의 보이지 않게 하고, 떠 있어야 하는 메뉴·팝오버·드래그 항목에만 분명한 그림자를 쓴다.
- 카드 구분을 그림자에만 의존하지 않는다.

### 8.4 편지지 메시지

개인 메시지는 매우 연한 초록 종이와 작은 하트 무늬를 사용한다. 닫힌 상태에서는 제목, 짧은 미리보기, 열기 힌트만 보이고 누르면 세로로 확장된다.

권장 본문:

> Music production—especially MIDI orchestration in FL Studio—gets better when we share the tools that make our workflows easier.  
> I’m starting with BRSO Articulate improvements and preset sharing, then expanding into music, development notes, and small tips & tricks along the way.

구현 원칙:

- 텍스트 프레임과 편지지 컨테이너는 내용에 따라 세로로 자란다.
- 텍스트를 고정 높이로 crop하지 않는다.
- 하트 패턴은 본문 가독성을 방해하지 않을 정도로 옅게 한다.
- 전체 편지 또는 명확한 헤더를 클릭해 열 수 있게 한다.
- `aria-expanded`, 키보드 조작, focus 상태를 제공한다.
- Figma 프로토타입에서는 닫힘/열림 variant와 Smart Animate를 사용한다.

### 8.5 BRSOfriend 아이콘

- 검은 단색, 투명 배경, 둥근 끝과 일정한 시각적 굵기를 유지한다.
- 음표 기호는 음악가가 한눈에 articulation 차이를 알아볼 수 있어야 한다.
- Spiccato, Staccato, Marcato, Sustain, Tremolo, Harmonic, Trills, Pizzicato, Trill, Run, Default의 묶음으로 관리한다.
- Default는 복잡한 쉼표 대신 단순한 가로선 `—` 모양을 사용한다.
- Spiccato는 Staccato보다 짧다는 차이가 직관적으로 보여야 하며 최종 도안은 별도 검토한다.
- 512×512 원본을 유지하고 UI에서는 동일한 optical size로 표시한다.

## 9. 도구 UI 원칙

### 9.1 BRSOfriend

- Kontakt와 BRSO Articulate 사이에서 반복되는 클릭을 줄이는 것이 최우선이다.
- 키스위치 이름, 노트, 색상, 아이콘을 한 행 또는 한 흐름에서 빠르게 편집하게 한다.
- C-2 기준 인접 다음 음 자동 설정과 Enter 기반 다음 항목 이동을 지원한다.
- 개별 `.fst`와 압축 폴더의 file tree는 구조가 한눈에 보이도록 한다.
- MIDI Port는 기본 흐름을 방해하지 않는 선택 설정으로 둔다.
- 위험한 덮어쓰기 전에 원본 보존과 변경 내용을 분명히 안내한다.

### 9.2 KontaktCutter / Audio Tokenizer

흐름을 `MIDI 생성 → Kontakt에서 렌더 → WAV 불러오기 → Tokenize → Normalize/검증 → Drag & Drop` 단계로 명확히 나눈다.

- 음역, 길이, 간격, velocity 설정은 음악 용어로 표시한다.
- 음 겹침과 0 dB 문제는 작업이 끝난 뒤가 아니라 발견 즉시 경고한다.
- 각 token의 name, brand, instrument, key, velocity 정보가 소리와 함께 유지되어야 한다.
- 긴 처리 작업은 진행률, 남은 단계, 취소 가능 여부를 보여 준다.

### 9.3 MixerPalette / Color Palette

- 악기군별 색을 빠르게 만들고 충돌을 비교하는 것이 핵심이다.
- 색상 자체뿐 아니라 이름, 악기군, 소유자, 공유 출처를 함께 보여 준다.
- 색각 차이가 있어도 구분할 수 있도록 텍스트와 패턴 또는 그룹 정보를 병행한다.

### 9.4 EffectChainShare

- 체인의 순서, 플러그인 이름, 주요 설정, 제작자와 기여일을 명확히 보여 준다.
- Contributors 프로필과 자연스럽게 연결한다.
- 공유와 홍보는 가능하되 출처와 권리 범위를 모호하게 만들지 않는다.

## 10. 모션과 귀여움 포인트

모션은 기능 완료를 설명하거나 캐릭터성을 드러내는 데 사용한다.

- 기본 UI 전환: 약 150–220ms
- 패널·편지 확장: 약 250–400ms
- 캐릭터 연출: 더 길 수 있지만 사용자의 진행을 막지 않는다.
- hover 이동은 2–4px 수준으로 제한한다.
- spring은 사카노, 리본, 로고처럼 유기적인 요소에만 선택적으로 사용한다.
- 저장 성공 시 작은 리본 흔들림, 빈 상태에서 사카노 등장 같은 보상을 고려한다.
- 계속 반복되는 장식 애니메이션은 피한다.

기능이 모두 안정된 후 귀여움 포인트를 추가한다. 귀여운 연출 때문에 편집 속도나 정보 판독성이 떨어지면 제거한다.

## 11. shadcn 사용 방침

shadcn은 AbelTools의 완성된 외형이 아니라 접근성과 상호작용을 위한 골격으로 사용한다.

적합한 영역:

- Dialog, Alert Dialog, Dropdown, Select, Tabs, Accordion
- Form, Tooltip, Toast, Sheet
- 키보드 동작과 focus 관리가 필요한 기본 컴포넌트

직접 설계할 영역:

- 메인 랜딩과 브랜드 섹션
- 리본·사카노·편지지 같은 고유 요소
- Piano Roll, keyswitch editor, file tree, waveform/token editor
- MixerPalette와 EffectChain 시각화

shadcn 기본 neutral 색상과 radius를 그대로 노출하지 않고 AbelTools의 토큰과 상태 정의를 입힌다.

## 12. 접근성과 사용성

- 일반 텍스트는 WCAG AA 수준의 명도 대비를 목표로 한다.
- focus ring을 없애지 않는다.
- 색 하나만으로 기능 종류나 상태를 구분하지 않는다.
- 클릭 영역은 가능하면 최소 44×44px을 확보한다.
- 아이콘에는 이름 또는 인접 레이블을 제공한다.
- 모든 핵심 기능은 키보드로 사용할 수 있어야 한다.
- 애니메이션 감소 설정을 존중한다.
- 한국어/영어 전환 시 레이아웃이 깨지지 않도록 고정 폭 텍스트 상자를 피한다.
- 번역 문자열의 인코딩은 UTF-8로 유지하고 깨진 한글을 출시하지 않는다.

## 13. Figma와 구현 규칙

- 색상, 간격, radius, type style은 Figma Variable/Style과 CSS token을 가능한 한 같은 이름으로 관리한다.
- 의미 기반 이름(`tool-accent`, `communication-surface`)을 우선하고 단순 색 이름은 원시 팔레트에만 쓴다.
- 컴포넌트는 Default, Hover, Pressed, Focus, Disabled 상태를 설계한다.
- 확장 가능한 컴포넌트는 고정 높이보다 Auto Layout/Hug contents를 우선한다.
- 이미지 원본과 실제 사용 크기를 구분하고, 로고 및 아이콘은 가능하면 SVG 또는 고해상도 투명 자산으로 관리한다.
- 장식 자산은 `public` 안에서 logo, character, pattern, tool-icon처럼 역할별로 구분한다.

## 14. 아직 결정하지 않은 사항

다음 항목은 이 문서에서 임의로 확정하지 않는다.

- 최종 색상 HEX/OKLCH 값
- 로고 워드마크의 최종 벡터와 최소 사용 크기
- 사카노의 최종 캐주얼 복장
- 한국어와 영문의 최종 본문 글꼴 조합
- EffectChainShare의 실제 공유 포맷과 권리 정책
- 모바일에서 복잡한 도구가 제공할 편집 범위
- 다크 모드 제공 여부

## 15. 디자인 검토 체크리스트

새 화면을 만들 때 아래 질문에 답한다.

- 이 화면은 기능(노랑), 소통(초록), 콘텐츠(중립) 중 어디에 속하는가?
- 가장 중요한 행동이 한눈에 보이는가?
- 귀여운 요소가 기능을 돕거나 브랜드 기억을 만드는가?
- 사카노나 리본을 빼도 정보 구조가 성립하는가?
- 모바일과 키보드에서도 핵심 흐름을 끝낼 수 있는가?
- 텍스트 길이가 늘어나도 잘리거나 겹치지 않는가?
- 기존 shadcn 화면처럼 보이는 것이 아니라 AbelTools처럼 보이는가?
- 음악가가 개발자 설명 없이 용어와 흐름을 이해할 수 있는가?
